# 账号池技术实现方案

## 1. 模块边界

```text
Views/AccountsView.swift
    -> Stores/AccountStore.swift
    -> Services/AccountService.swift
       ├─ KeychainStore.swift       token
       ├─ OAuth/refresh             PKCE 登录与 token 刷新
       ├─ usage/warmup               额度查询与预热
       └─ auth.json switch           当前账号切换
```

现有 `AppStore` 继续负责历史/备份页面；账号池单独使用 `AccountStore`，避免把认证状态塞进历史记录状态机。

## 2. 持久化

### Keychain

使用 Security.framework 的 generic password item：

- service：`com.hanshuheng.CodexHistorySync.account`
- account：账号池内部 UUID
- value：Codable 的 `AccountCredentials` JSON

`AccountCredentials` 包含 access token、refresh token、id token、ChatGPT account ID、workspace ID 和 token 更新时间。Keychain 读写错误直接返回，不把 token 写入错误文本。

### JSON

路径：`~/.codex/account_pool.json`，权限设置为 `0600`。

只保存以下非敏感字段：账号 ID、显示名、邮箱、套餐、状态、额度快照、最近刷新时间、最近错误、Keychain 引用。写入使用临时文件 + atomic replace。

## 3. 登录流程

1. `OAuthService.startLogin()` 生成 64 字节 PKCE verifier、challenge 和 32 字节 state。
2. 用已验证的 OpenAI Codex OAuth client ID、issuer、redirect URI 构建授权地址。
3. 启动只监听 loopback 的 localhost callback server，使用 `NSWorkspace` 打开系统浏览器，回调仅接受本次 state。
4. 用 authorization code + verifier 调 token endpoint。
5. 解析 id token claim，提取账号 ID、workspace、邮箱和套餐；无法解析时保留账号但标为“身份信息不完整”。
6. credentials 写 Keychain，metadata 写 JSON，随后立即执行一次额度刷新。

如果系统回调能力不足，退回 localhost callback server；该 server 只监听 loopback、验证 state、成功后立即关闭。

## 4. token refresh

- access token 过期或 usage 返回 401 时，使用 refresh token 调 token endpoint。
- 成功后覆盖 Keychain credentials，再重试原 usage/warmup 请求一次。
- refresh token 失效时账号状态改为 `reauthRequired`，不删除已有快照。
- 所有 URL、header、错误 body 在日志中脱敏。

## 5. 额度接口

`UsageService` 使用每账号独立 token：

- GET Codex usage endpoint；
- 设置 `Authorization: Bearer`、`ChatGPT-Account-ID` 和 workspace 相关 header；
- 解析主窗口和次窗口；
- 将 `usedPercent` 转换为 `remainPercent = 100 - usedPercent`；
- 将 reset timestamp 转为 `Date`，窗口长度保留分钟；
- 单账号失败返回 `AccountUsageSnapshot(error:)`，不影响其他账号。

刷新策略：页面手动刷新优先；登录后自动刷新一次；预热成功后再次刷新；不实现后台定时轮询，避免无请求需求时消耗额度。

## 6. 预热接口

`WarmupService.warmup(accountIDs:message:)`：

1. 空 ID 集合展开为所有账号；
2. 默认 message=`hi`，model 使用当前 Codex 默认模型或固定兼容模型；
3. POST `https://chatgpt.com/backend-api/codex/responses`，使用单条最小输入；
4. 账号级 timeout，失败时 refresh token 并重试一次；
5. 成功后调用 `UsageService.refresh(accountID:)`；
6. 返回逐账号结果和汇总计数。

预热不会写 reset 时间。5 小时窗口是否刷新由服务端决定，客户端只显示服务端返回值。

## 7. 账号切换

`AccountSwitcher.switchTo(accountID:)`：

1. 从 Keychain 读取目标 credentials；
2. 检查 access token 非空；
3. 将当前 `~/.codex/auth.json` 复制到 `~/.codex/account_pool_backups/auth-<timestamp>.json`；
4. 按 Codex-Manager `build_direct_auth_json` 的结构写入 `OPENAI_API_KEY: null`、`tokens.id_token/access_token/refresh_token/account_id` 和 UTC `last_refresh`；
5. 使用 `Data.write(options: .atomic)` 写入；
6. 重新读取并校验 JSON 结构，失败则恢复备份；
7. 更新当前账号标记，提示重启 Codex 或重新连接。

切换不杀死 Codex 进程、不修改历史数据库、不尝试热替换已运行会话的内存 token。

## 8. UI

在侧边栏新增“账号池”，主区使用原生 `Table`：账号、套餐、5 小时额度、7 天额度、状态、操作。工具栏提供“登录账号”“刷新全部”“预热”“切换当前”。额度用剩余百分比和 reset 时间双重显示，未知值显示“未知”。

每个危险动作提供可见状态：登录中、刷新中、预热中、切换确认、失败原因。token 不在表格或 alert 中显示。

## 9. 验证

- `swift build`
- 原生 service 自检：临时目录、JSON 持久化、Keychain 错误处理和 auth 切换回滚
- 手动检查：两账号登录、额度刷新、预热逐条结果、切换后 `auth.json` 结构
- `codegraph sync .`
- `./script/build_and_run.sh --verify`

网络真实调用必须由用户已登录账号执行；没有真实账号时使用 URLProtocol/mock 响应测试 token refresh、额度解析和预热结果聚合。

## 10. 风险

- OpenAI OAuth、usage endpoint 和 Codex auth.json 字段可能变更；集中在 `OAuthService`、`UsageService` 和 `CodexAuthFile`，便于更新。
- 预热会消耗账号一次请求额度，UI 必须明确提示。
- 当前进程可能缓存旧 token，切换后仍需重启/重新连接 Codex。
- Keychain item 删除或 refresh token 失效时，只能要求重新登录，不能从快照恢复凭据。
