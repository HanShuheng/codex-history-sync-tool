# 账号池技术实现方案

## 1. 模块边界

```text
Views/AccountsView.swift
    -> Stores/AccountStore.swift
    -> Services/AccountService.swift
       ├─ AccountModels.swift       账号、凭据和 profile 备份模型
       ├─ AppPaths.swift            Codex 目录与本工具目录分离
       ├─ LocalConfigStore.swift    本工具 config.json 的并发读写
       ├─ AccountUsageParser.swift  5 小时/7 天窗口解析
       ├─ CodexAuthFile.swift       直连 auth.json 结构
       ├─ OAuth/refresh             PKCE 登录与 token 刷新
       ├─ usage/warmup               额度查询与预热
       └─ auth.json switch           当前账号切换
```

现有 `AppStore` 继续负责历史/备份页面；账号池单独使用 `AccountStore`，避免把认证状态塞进历史记录状态机。

## 2. 持久化

### JSON

路径：`~/.codexhistorysync/config.json`，权限设置为 `0600`；目录权限为 `0700`。

文件结构：

```json
{
  "accounts": [
    {
      "id": "account-id",
      "displayName": "账号显示名",
      "credentials": {
        "idToken": "...",
        "accessToken": "...",
        "refreshToken": "...",
        "accountID": "...",
        "workspaceID": "..."
      }
    }
  ],
  "selected_thread_ids": ["thread-id"]
}
```

`accounts` 保存账号元数据、额度快照和 credentials；`selected_thread_ids` 保存历史勾选。文件不存在时两个字段都使用空集合，历史记录首次打开全部不勾选。所有更新必须先读入完整配置、只修改自己的字段，再使用临时文件 + atomic replace 写回；账号保存和历史选择保存共用进程内锁，避免互相覆盖。

`~/.codex` 只作为 Codex 原始目录：读取 `config.toml`、数据库、sessions、global state，并在直连切换时写入 `auth.json`。本工具自己的配置、选择和备份禁止写入 `~/.codex`。

## 3. 登录流程

1. `OAuthService.startLogin()` 生成 64 字节 PKCE verifier、challenge 和 32 字节 state。
2. 用已验证的 OpenAI Codex OAuth client ID、issuer、redirect URI 构建授权地址。
3. 启动只监听 loopback 的 localhost callback server，使用 `NSWorkspace` 打开系统浏览器，回调仅接受本次 state。
4. 用 authorization code + verifier 调 token endpoint。
5. 解析 id token claim，提取账号 ID、workspace、邮箱和套餐；无法解析时保留账号但标为“身份信息不完整”。
6. credentials 和 metadata 一起写入 `config.json.accounts`，随后立即执行一次额度刷新。

如果系统回调能力不足，退回 localhost callback server；该 server 只监听 loopback、验证 state、成功后立即关闭。

## 4. token refresh

- access token 过期或 usage 返回 401 时，使用 refresh token 调 token endpoint。
- 成功后覆盖本地配置文件中的 credentials，再重试原 usage/warmup 请求一次。
- refresh token 失效时账号状态改为 `reauthRequired`，不删除已有快照。
- 所有 URL、header、错误 body 在日志中脱敏。

## 5. 额度接口

`UsageService` 使用每账号独立 token：

- GET Codex usage endpoint；
- 设置 `Authorization: Bearer`、`ChatGPT-Account-ID` 和 workspace 相关 header；
- 解析主窗口和次窗口；
- 将 `usedPercent` 转换为 `remainPercent = 100 - usedPercent`；
- 将 reset timestamp 转为 `Date`，窗口长度保留分钟；
- 单账号失败返回 `AccountUsageSnapshot(error:)`，不影响其他账号。

刷新策略：页面手动刷新优先；登录后自动刷新一次；预热成功后再次刷新；不实现后台定时轮询，避免无请求需求时消耗额度。

## 6. 预热接口

`WarmupService.warmup(accountIDs:message:)`：

1. 空 ID 集合展开为所有账号；
2. 默认 message=`hi`，失败后用 `你好` 重试一次；先按账号调用 `https://chatgpt.com/backend-api/codex/models?client_version=...`，过滤不可用模型并按 Codex-Manager 的目录顺序选择第一项；目录请求失败时回退当前 Codex `config.toml` 的 `model`，但目录明确返回“没有可用模型”时不回退；两者都没有时停止本账号预热并提示无法确定可用模型；
3. POST `https://chatgpt.com/backend-api/codex/responses`，使用单条最小输入；
4. 账号级 timeout，SSE 接受 `response.completed`、`response.done`、`[DONE]`；`error/response.failed/response.incomplete` 返回具体错误；401/403 时 refresh token 并重试一次；
5. 成功后调用 `UsageService.refresh(accountID:)`；
6. 返回逐账号结果和汇总计数。

预热不会写 reset 时间。5 小时窗口是否刷新由服务端决定，客户端只显示服务端返回值。

预热成功判定还要检查额度：Responses SSE 收到完成事件只代表请求完成；预热后的 usage 若 5 小时或 7 天剩余额度为 0，则返回“当前账号没有可用额度”，不显示预热成功。

模型目录接口没有稳定公开的 token 单价/倍率字段，因此本项目不会伪造“最低 token”判断；实际采用 Codex-Manager 相同的可用性过滤和目录排序。普通 OpenAI API 的 `GET /v1/models` 只适用于 API Key 模式，不能替代 ChatGPT/Codex OAuth 账号的 Codex 专用目录。

## 7. 账号切换：严格对齐 Codex-Manager

`AccountSwitcher.switchTo(accountID:)`：

1. 从 `~/.codexhistorysync/config.json.accounts` 读取目标 credentials；
2. 检查账号存在、账号状态为 active、access token 非空；
3. 若 refresh token 存在，先调用 token endpoint 刷新并把新 token 写回 `config.json`；
4. 如果 `~/.codexhistorysync/account_pool_backups/codex-profile.json` 不存在，保存当前 `~/.codex/auth.json` 和 `~/.codex/config.toml` 原文；后续切换不覆盖原始备份；
5. 按 `build_direct_auth_json` 结构生成 `OPENAI_API_KEY: null`、`tokens.id_token/access_token/refresh_token/account_id`、UTC `last_refresh`；
6. 仅删除 `model_provider = "cm"` 和 `[model_providers.cm]`，保留 `model` 和其他 provider；没有 `cm` 时保持 config 内容不变；
7. 原子写入 `~/.codex/auth.json` 和 `~/.codex/config.toml`，再重新解析 auth JSON，校验 access token 与目标 token 完全一致；
8. 任意写入或校验失败，按本次切换前的内存快照恢复两个文件；原始 profile 备份仍保留；
9. 更新 `config.json.accounts[].isCurrent`，提示重启 Codex 或重新连接。

### 7.1 为什么要同时改 config.toml

Codex-Manager 的 direct mode 会移除它自己注册的 `cm` provider，否则 Codex 仍可能继续走本地网关。它不会删除其他 provider，也不会把任意 `model_provider` 无条件清空。本项目必须沿用这个边界：只清理自己管理的 provider 标识，避免破坏用户自己的 provider 配置。

### 7.2 失败和恢复边界

- token 刷新失败：不写 auth/config，不改变当前账号；界面显示重新登录。
- 原始备份失败：不写 auth/config，直接返回错误。
- auth 写成功、config 写失败：恢复 auth 和 config 的切换前快照。
- auth 校验失败：恢复两个文件，并保留原始 profile 备份用于人工恢复。
- 切换成功但 Codex 已运行：不尝试修改进程内 token，只提示重启/重新连接。

切换不杀死 Codex 进程、不修改历史数据库、不尝试热替换已运行会话的内存 token。

切换后的历史同步：账号池表格上方的开关默认关闭，状态保存到 `~/.codexhistorysync/config.json` 的 `auto_sync_after_account_switch`。开启后，切换写入 profile 成功时调用 `HistoryService.syncAll`，同步数据库中的全部历史记录（包括归档记录），不依赖历史页面勾选项。同步失败不会回滚已经成功的账号切换，会提示用户到历史记录页面手动同步。

## 8. UI

在侧边栏新增“账号池”，主区使用原生 `Table`：账号、套餐、5 小时额度、7 天额度、状态、操作。工具栏提供“登录账号”“刷新全部”“预热”“切换当前”。额度用剩余百分比和 reset 时间双重显示，未知值显示“未知”。

每个危险动作提供可见状态：登录中、刷新中、预热中、切换确认、失败原因。token 不在表格或 alert 中显示。

## 9. 验证

- `swift build`
- `./script/test_account_pool.sh`
- 原生 service 自检：临时目录、单文件配置合并写入、首次选择为空、direct config 只移除 `cm`、auth/config 双文件回滚
- 手动检查：两账号登录、额度刷新、预热逐条结果、切换后 `auth.json` 结构
- `codegraph sync .`
- `./script/build_and_run.sh --verify`

网络真实调用必须由用户已登录账号执行；没有真实账号时使用 URLProtocol/mock 响应测试 token refresh、额度解析和预热结果聚合。

## 10. 风险

- OpenAI OAuth、usage endpoint 和 Codex auth.json 字段可能变更；集中在 `OAuthService`、`UsageService` 和 `CodexAuthFile`，便于更新。
- 预热会消耗账号一次请求额度，UI 必须明确提示。
- 当前进程可能缓存旧 token，切换后仍需重启/重新连接 Codex。
- 本地凭据文件损坏或 refresh token 失效时，只能要求重新登录，不能从快照恢复凭据。
